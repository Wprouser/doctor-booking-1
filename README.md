# MediBook — Doctor Appointment Booking

A doctor appointment booking app built with React + Vite + TypeScript on the frontend and Express + PostgreSQL (Neon) on the backend, with authentication via [Better Auth](https://www.better-auth.com). Patients sign up, search doctors by specialization, view profiles, book appointments, and manage (view/cancel) their own bookings.

## Features

- Email + password sign up / sign in / sign out (Better Auth)
- Search doctors by name or specialization, with a specialization filter
- View a doctor's profile: bio, education, languages, rating, location, fee
- Sign-in required to book; booking a slot ties the appointment to your account
- View upcoming, past, and cancelled appointments — scoped to the signed-in user
- Cancel an upcoming appointment with a confirmation step
- Responsive layout (mobile through desktop) with light/dark support

## Tech stack

- **Frontend:** React 19 + TypeScript, Vite, Tailwind CSS v4, react-router-dom
- **Backend:** Node.js + Express, `pg` (node-postgres), Better Auth
- **Database:** Neon (serverless Postgres)

## Getting started

1. Create a [Neon](https://neon.tech) project and copy its connection string.
2. Copy `.env.example` to `.env` and fill in `DATABASE_URL` and `BETTER_AUTH_SECRET`:

   ```bash
   cp .env.example .env
   # generate a secret for BETTER_AUTH_SECRET:
   node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
   ```

3. Install dependencies, then set up the database:

   ```bash
   npm install

   # Better Auth's own tables (user, session, account, verification)
   npx @better-auth/cli migrate --config server/auth.ts

   # this app's tables (doctors, appointments) + seed mock doctors
   npm run db:migrate
   npm run db:seed
   ```

4. Run the app in development (Vite frontend + Express API together, with `/api` proxied to the backend):

   ```bash
   npm run dev
   ```

   Open the URL Vite prints (typically `http://localhost:5173`).

## Production build (single deployable app)

```bash
npm run build   # type-checks and builds the frontend into dist/
npm start       # Express serves dist/ and the /api/* routes on one port
```

One Node process serves both the static frontend and the API, so deployment is just "run `npm start`" behind a single port — no separate frontend host needed. Set `DATABASE_URL`, `BETTER_AUTH_SECRET`, and `BETTER_AUTH_URL` (the deployed public URL) in the deployment environment.

## Project structure

```
server/
  index.ts        Express app entry — mounts Better Auth + /api routes, serves dist/ in prod
  auth.ts         Better Auth config (email/password, Postgres via the pg Pool)
  db.ts           pg Pool (uses DATABASE_URL)
  db/
    schema.sql    App table definitions (doctors, appointments)
    migrate.ts    Runs schema.sql against the configured database
    seed.ts       Seeds the doctors table with mock data
    better-auth_migrations/  SQL generated by the Better Auth CLI (user/session/account/verification)
  lib/slots.ts    Deterministic per-doctor available-slot generation
  routes/         doctors.ts, appointments.ts — REST handlers (appointments require a session)

src/
  components/     Reusable UI components (DoctorCard, SlotPicker, Modal, ProtectedRoute, etc.)
  context/        AppointmentsContext — fetches/mutates appointments via the API
  data/           Static specialization list (doctor data lives in Postgres)
  lib/            api.ts (fetch wrappers), auth-client.ts (Better Auth React client), date/avatar helpers
  pages/          Route-level pages (Search, Doctor Profile, My Appointments, Sign In, Sign Up)
  types.ts        Shared TypeScript types
```

## API

| Method | Path                           | Auth required | Description                                          |
| ------ | ------------------------------ | -------------- | ----------------------------------------------------- |
| *      | `/api/auth/*`                   | —               | Better Auth (sign up, sign in, sign out, session, ...) |
| GET    | `/api/doctors`                   | No              | List doctors; supports `?q=` and `?specialization=`   |
| GET    | `/api/doctors/:id`                | No              | Doctor detail, including computed available slots     |
| GET    | `/api/appointments`               | Yes             | Appointments booked by the signed-in user             |
| POST   | `/api/appointments`               | Yes             | Book an appointment                                    |
| PATCH  | `/api/appointments/:id/cancel`    | Yes             | Cancel one of your own appointments                    |

## Data & persistence

Everything lives in Neon Postgres: Better Auth owns the `user`, `session`, `account`, and `verification` tables; this app owns `doctors` and `appointments` (`appointments.user_id` references `user.id`, so bookings are tied to a real account). There is no browser-side persistence anymore — signing out and back in (even on another device) shows the same appointments.
